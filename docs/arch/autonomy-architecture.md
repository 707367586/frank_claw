## 1. AI 自主性发展现状分析

### 1.1 自主性演进阶梯

AI Agent 的自主性可以划分为 5 个层级，每个层级代表不同的能力边界：

```
Level 5: 完全自主 (Full Autonomy)
         自主设定目标、自主学习新能力、跨域迁移
         ◈ 当前技术尚未达到

Level 4: 协作自主 (Collaborative Autonomy)  ← ClawX 目标层级
         多 Agent 协作、层级规划与分解、自我反思与纠错
         自主发现并学习新工具、理解物理环境
         ◈ 2025-2026 前沿: Claude Computer Use, OpenAI Operator,
           Devin, Multi-Agent Systems (AutoGen/CrewAI)

Level 3: 工具自主 (Tool Autonomy)
         ReAct 推理循环、多步工具调用、上下文感知决策
         ◈ 2024-2025 主流: Claude Tool Use, GPT Actions, LangChain Agents

Level 2: 记忆自主 (Memory Autonomy)
         持久化记忆、个性化适配、跨会话连续性
         ◈ ClawX 已有完整设计 (三层记忆架构)

Level 1: 响应式 (Reactive)
         单轮问答、无状态、无工具调用
         ◈ 传统 Chatbot

当前 ClawX 架构位于 Level 2-3 之间，本文档将架构提升至 Level 4
```

## 3. ReAct 推理循环 (Reasoning + Acting)

### 3.1 设计原理

传统的 ClawX 对话流是线性的：`用户输入 → Prompt 组装 → LLM 调用 → 返回`。这限制了 Agent 处理需要多步骤推理的复杂任务。

ReAct 循环让 Agent 具备**迭代推理**能力：

```
┌──────────────────────────────────────────────────────────────────────┐
│                      ReAct Loop Architecture                          │
│                                                                       │
│  用户输入: "帮我调研竞品 X 的定价策略，并生成对比分析报告"              │
│      │                                                                │
│      ▼                                                                │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Iteration 1                                                     │ │
│  │  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌────────────────┐  │ │
│  │  │ Thought │──▶│ Action  │──▶│Observ.  │──▶│ Evaluation     │  │ │
│  │  │ 我需要先│   │ 使用    │   │ 搜索到  │   │ 信息不够完整   │  │ │
│  │  │ 搜索竞品│   │ web_    │   │ 3 篇文章│   │ 需要更多细节   │  │ │
│  │  │ 定价信息│   │ search  │   │ 关于X   │   │ → 继续迭代     │  │ │

│  └──────────────────────────────────────────────────────────────────┘ │

│  │  │ 需要读取│   │ read_   │   │ 获取到  │   │ 定价数据已充分 │  │ │

### 3.2 ReAct 状态机

```rust
/// ReAct 循环的核心状态机
pub enum ReActState {
    /// 思考阶段: LLM 分析当前状态，决定下一步
    Thinking {
        context: ThinkingContext,
        iteration: u32,
    },
    /// 行动阶段: 执行 LLM 决定的工具调用
    Acting {
        action: PlannedAction,
        iteration: u32,
    },
    /// 观察阶段: 收集工具执行结果
    Observing {
        action_result: ActionResult,
        iteration: u32,
    },
    /// 评估阶段: 判断是否需要继续迭代
    Evaluating {
        accumulated_context: Vec<ReActStep>,
        iteration: u32,
    },
    /// 完成: 返回最终结果
    Completed {
        result: AgentResponse,
        steps: Vec<ReActStep>,
        total_iterations: u32,
    },
    /// 需要用户确认 (高风险操作)
    AwaitingConfirmation {
        pending_action: PlannedAction,
        risk_level: RiskLevel,
        explanation: String,
    },
}

/// 单步 ReAct 记录 (用于推理链展示)
pub struct ReActStep {
    pub thought: String,          // Agent 的思考过程
    pub action: PlannedAction,    // 决定执行的操作
    pub observation: String,      // 操作结果
    pub evaluation: StepEval,     // 评估结果
    pub duration_ms: u64,
    pub tokens_used: u32,
}

/// 步骤评估
pub enum StepEval {
    Continue { reason: String },         // 需要继续
    Complete { confidence: f64 },        // 任务完成
    NeedHumanInput { question: String }, // 需要用户输入
    Error { recoverable: bool, msg: String },
}
```

### 3.3 安全围栏 (Safety Guardrails)

```
ReAct 循环安全约束:

┌──────────────────────────────────────────────────────────┐
│                    Safety Guardrails                       │
│                                                           │
│  1. 最大迭代次数                                          │
│     ├── 默认: max_iterations = 10                        │
│     ├── 用户可配置: 1-25                                  │
│     └── 每次迭代消耗 Token 追踪，超过预算自动终止          │
│                                                           │
│  2. 操作风险分级                                          │
│     ├── Low: 纯读取操作 (search, read) → 自动执行         │
│     ├── Medium: 受限写入 (write_to_workspace) → 自动执行  │
│     ├── High: 外部交互 (send_message, api_call) → 需确认  │
│     └── Critical: 系统操作 (exec_shell, delete) → 必须确认│
│                                                           │
│  3. 循环检测                                              │
│     ├── 连续 3 次执行相同 Action → 告警并暂停              │
│     ├── 检测到无进展 (observation 重复) → 请求用户指导     │
│     └── Token 总消耗 > budget * 0.8 → 警告并建议总结      │
│                                                           │
│  4. 实时透明                                              │
│     ├── 每步 Thought 实时流式展示给用户                    │
│     ├── 用户可随时按 Esc 中断循环                          │
│     └── 中断后保留已完成步骤的结果                         │
└──────────────────────────────────────────────────────────┘
```

## 4. 自我反思与纠错 (Self-Reflection)

### 4.1 设计原理

自我反思是最能带来 "Aha Moment" 的能力——用户看到 Agent 不仅给出答案，还能自己检查、发现问题、并主动修正。这种类人行为大幅提升用户信任感。

### 4.2 反思架构

### 4.3 反思类型

```rust
/// 反思检查类型
pub enum ReflectionCheck {
    /// 输出质量: 完整性、准确性、格式
    QualityCheck {
        completeness: f64,   // 是否完整回答了用户问题
        accuracy: f64,       // 事实准确性 (可通过 KB 验证)
        format_match: f64,   // 是否符合用户偏好的格式
    },

    /// 一致性检查: 与记忆、知识库、历史的一致性
    ConsistencyCheck {
        memory_consistent: bool,     // 与用户记忆一致
        kb_consistent: bool,         // 与知识库内容一致
        self_consistent: bool,       // 前后逻辑自洽
        contradictions: Vec<String>, // 发现的矛盾点
    },

    /// 安全检查: 是否包含有害内容
    SafetyCheck {
        has_sensitive_data: bool,
        has_harmful_content: bool,
        follows_guidelines: bool,
    },

    /// 任务完成度: 是否完成了用户的实际需求
    TaskCompletionCheck {
        original_intent: String,     // 用户原始意图
        addressed_aspects: Vec<String>,
        missing_aspects: Vec<String>,
        completion_ratio: f64,
    },
}

/// 反思结果
pub struct ReflectionResult {
    pub overall_confidence: f64,
    pub checks: Vec<ReflectionCheck>,
    pub suggested_improvements: Vec<Improvement>,
    pub should_revise: bool,
    pub revision_strategy: Option<RevisionStrategy>,
}

/// 修正策略
pub enum RevisionStrategy {
    /// 补充遗漏内容
    Supplement { missing: Vec<String> },
    /// 修正错误
    Correct { errors: Vec<Correction> },
    /// 重新组织格式
    Reformat { target_format: String },
    /// 完全重做
    Redo { reason: String },
    /// 请求用户澄清
    Clarify { questions: Vec<String> },
}
```

### 4.4 反思触发策略

并非每次都触发反思（LLM 开销过大），采用智能触发：

| 触发条件 | 反思深度 | 说明 |
|---------|---------|------|
| 用户明确要求高质量 ("仔细"、"确保准确") | Full | 全量检查 |
| Agent 输出包含数据/数字/事实声明 | Accuracy | 数据验证 |
| 输出长度 > 500 字 | Completeness | 完整性检查 |
| 涉及用户偏好的任务 (写作风格等) | Consistency | 记忆一致性 |
| ReAct 循环中每次关键决策点 | Quick | 快速自检 |
| 对话轮次 > 5 且任务复杂 | TaskCompletion | 任务完成度 |

## 6. 多 Agent 协作编排 (Multi-Agent Orchestration)

### 6.1 设计原理

ClawX 已支持多 Agent 配置，但 v4.0 架构中 Agent 之间是完全隔离的（循环守卫 L9 防止 Agent 间调用）。本设计引入**受控的结构化协作**，允许 Agent 在明确的协作协议下互相委托任务。

> **关键区别**：不是无限制的 Agent A ↔ B 互调（仍然危险），而是通过 **Orchestrator 模式** 实现单向、可控的委托。

### 6.2 协作模式

```

### 6.3 协作协议

```rust
/// Agent 间协作请求
pub struct DelegationRequest {
    pub id: DelegationId,
    pub from_agent: AgentId,             // 发起方
    pub to_agent: AgentId,               // 被委托方
    pub pattern: CollaborationPattern,   // 协作模式
    pub task: DelegatedTask,             // 委托任务
    pub context: DelegationContext,      // 提供给被委托方的上下文
    pub constraints: DelegationConstraints,
    pub created_at: DateTime<Utc>,
}

/// 委托上下文 (最小化，不暴露发起方完整记忆)
pub struct DelegationContext {
    pub task_description: String,        // 任务描述
    pub relevant_data: Vec<DataSnippet>, // 相关数据片段
    pub output_format: Option<String>,   // 期望输出格式
    pub max_tokens: u32,                 // Token 预算
}

/// 委托约束
pub struct DelegationConstraints {
    pub max_depth: u32,                  // 最大委托链深度 (默认 2)
    pub max_iterations: u32,             // 被委托方最大迭代次数
    pub timeout_secs: u64,               // 超时时间
    pub allowed_tools: Vec<ToolId>,      // 被委托方可用工具
    pub require_user_approval: bool,     // 是否需要用户批准委托
}

/// 协作模式
pub enum CollaborationPattern {
    Delegation,                // 委托: 1:1 任务下发
    Negotiation { agents: Vec<AgentId> }, // 协商: 多方意见汇总
    Pipeline { stages: Vec<PipelineStage> }, // 流水线: 顺序加工
}
```

### 6.4 与现有安全架构的集成

```
协作安全层 (扩展 L9 循环守卫):

现有 L9: 检测 Agent A ↔ B 乒乓模式，直接拒绝
升级 L9':
  ├── 委托链追踪: 每个 DelegationRequest 携带 chain_hash
  ├── 深度限制: depth > max_depth → 拒绝
  ├── 环路检测: 调用链中出现同一 Agent → 拒绝
  ├── 频率限制: 单 Agent 每小时委托次数上限
  ├── 上下文隔离: 被委托方只看到 DelegationContext，不看到完整记忆
  └── 审计日志: 所有协作行为记入 audit_log，可追溯
```

│  │    1. 截屏 → LLM 分析当前状态                                 │    │

### 8.3 三层操作策略

Computer Use 不应总是用"截屏+点击"这种重型方式，而是智能选择最高效的方法：

```

│  示例:                                                    │

│                                                           │
│  优势: 快速、精确、不受 UI 变化影响                        │
├───────────────────────────────────────────────────────────┤

│  ─────────────────────────────                            │

│                                                           │
│  示例:                                                    │

## 9. 信任渐进系统 (Progressive Trust)

### 9.1 设计原理

自主性越高，风险越大。信任渐进系统让 Agent 通过成功执行逐步积累用户信任，用信任等级动态调整 Agent 的自主权限边界。

### 9.2 信任等级模型

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Progressive Trust Model                             │
│                                                                       │
│  Trust Level 0: 新手 (Novice)                                        │
│  ─────────────────────────────                                        │
│  • 所有工具调用需用户确认                                              │
│  • ReAct 循环限制 3 次迭代                                            │
│  • 无 Computer Use                                                    │
│  • 无 Agent 间委托                                                    │
│  • 触发条件: 新创建的 Agent                                           │
│                                                                       │
│  Trust Level 1: 学徒 (Apprentice)                                     │
│  ─────────────────────────────────                                    │
│  • 低风险操作自动执行 (读取、搜索)                                    │
│  • ReAct 循环限制 5 次迭代                                            │
│  • Computer Use: 只读模式                                             │
│  • 无 Agent 间委托                                                    │
│  • 升级条件: 成功执行 10 个任务，0 次严重错误                          │

│                                                                       │
│  Trust Level 3: 伙伴 (Partner)                                        │
│  ─────────────────────────────                                        │

│  • Computer Use: 白名单应用自主操作                                   │

│                                                                       │

└──────────────────────────────────────────────────────────────────────┘

### 9.3 信任评分算法

```rust
/// 信任评分计算
pub struct TrustScore {
    pub level: TrustLevel,
    pub score: f64,                   // 0-100 综合分
    pub successful_tasks: u64,        // 成功任务数
    pub total_tasks: u64,             // 总任务数
    pub user_rollbacks: u64,          // 用户回滚次数
    pub user_corrections: u64,        // 用户纠正次数
    pub positive_feedback: u64,       // 正面反馈
    pub negative_feedback: u64,       // 负面反馈
    pub last_incident: Option<DateTime<Utc>>, // 最近一次严重事件
}

impl TrustScore {
    /// 综合评分计算
    pub fn calculate(&self) -> f64 {
        let success_rate = self.successful_tasks as f64 / self.total_tasks.max(1) as f64;
        let rollback_penalty = self.user_rollbacks as f64 * 5.0;
        let correction_penalty = self.user_corrections as f64 * 2.0;
        let feedback_bonus = (self.positive_feedback as f64 - self.negative_feedback as f64 * 3.0).max(0.0);
        let recency_factor = self.last_incident
            .map(|t| {
                let days = (Utc::now() - t).num_days() as f64;
                (days / 30.0).min(1.0) // 30 天后事件影响消退
            })
            .unwrap_or(1.0);

        let raw = success_rate * 60.0
            + feedback_bonus.min(20.0)
            - rollback_penalty
            - correction_penalty;

        (raw * recency_factor).clamp(0.0, 100.0)
    }
}
```

### 9.4 信任降级 (Demotion)

信任可以降级，严重错误导致即时降级：

| 事件 | 影响 |
|------|------|
| 用户回滚 Agent 操作 | score -= 5 |
| 用户纠正 Agent 输出 | score -= 2 |
| Agent 执行出错 (可恢复) | score -= 3 |
| Agent 执行出错 (不可恢复) | 立即降级一层 |
| 安全事件 (DLP 触发等) | 立即降级到 Level 0 |
| 用户明确表示不满 | score -= 10 |

## 10. 预测性主动 (Predictive Proactivity)

### 10.1 设计原理

超越 PRD 中简单的 Cron 定时触发，实现**基于用户行为模式的预测性主动**——Agent 学习用户的工作节奏，在用户需要之前就准备好。

### 10.2 预测引擎

## 12. 自主性 Aha Moment 场景设计

### 12.1 高价值场景一览

| # | 场景 | 触发 | Agent 自主行为 | Aha Moment |
|---|------|------|--------------|-----------|
| 1 | **自主纠错** | 用户要求写代码 | Agent 写完代码后自检发现 bug，主动修复并说明 | "它居然能自己发现自己的错误！" |
| 2 | **主动准备** | 日历显示 30 分钟后有会议 | Agent 自动整理相关资料并推送 | "它知道我马上要开会！" |
| 3 | **跨 Agent 协作** | 用户要求 PM Agent 完成复杂项目 | PM 自动委托任务给 Dev 和 Writer Agent | "它们居然自己分工协作！" |
| 4 | **操作电脑** | "帮我把这些照片导入 Lightroom" | Agent 打开 Lightroom，操作导入界面 | "它居然在帮我操作软件！" |
| 5 | **学习偏好** | 用户多次修改 Agent 的输出格式 | Agent 主动说"我注意到你偏好 X 格式，已记住" | "它学会了我的习惯！" |
| 6 | **深度研究** | "帮我深入调研 XX 技术" | Agent 自主执行 8 步 ReAct：搜索→阅读→分析→补充→对比→总结→生成报告 | "它像真的研究员一样工作！" |
| 7 | **预判需求** | 用户打开了项目代码文件 | Agent 检测到文件变更，主动提供相关文档和最近的修改记录 | "它知道我接下来要做什么！" |
| 8 | **异常告警** | 磁盘空间突然降到 5% | Agent 分析原因，发现是 Docker 镜像占用，建议清理 | "它比我更早发现问题！" |

### 12.2 Aha Moment 度量

| 指标 | 目标 | 说明 |
|------|------|------|
| 首次 Aha Moment 时间 | < 10 分钟使用 | 新用户应在 10 分钟内体验到至少 1 个 Aha Moment |
| 日均 Aha Moment | >= 2 次 | 活跃用户每天至少感受 2 次惊喜 |
| 自主纠错触发率 | 输出的 15-25% | 太低没感知，太高说明初始质量差 |
| 预测性建议采纳率 | >= 60% | 低于 60% 说明预测不准，需调优 |
| 信任等级提升速率 | 1 周内达到 Level 2 | 正常使用 1 周后应有足够自主权限 |

## 13. 架构决策记录 (ADR)

### ADR-020: 自主性能力集成在 clawx-runtime 而非独立 Crate

**状态:** 已采纳

**背景:** 自主性引擎 (ReAct, Planner, Reflection 等) 需要频繁访问 LLM、Memory、Security 等模块。

**决策:** 作为 clawx-runtime 的子模块实现，而非独立 Crate。

**理由:**
- 自主性引擎是运行时编排的核心部分，与 Runtime 紧密耦合
- 独立 Crate 会引入大量跨模块调用开销
- 通过子模块保持代码组织清晰度

### ADR-021: 采用 ReAct 而非 Plan-and-Execute 作为基础推理模式

**状态:** 已采纳

**背景:** 两种主流 Agent 推理模式——ReAct (逐步推理) vs Plan-and-Execute (先全量规划再执行)。

**决策:** ReAct 为基础推理循环，Planner 作为 ReAct 内部的可选增强。

**理由:**
- ReAct 更灵活，可根据中间结果动态调整
- Plan-and-Execute 在信息不完整时容易产生错误计划
- 两者组合：复杂任务先 Plan 再在每步中 ReAct，简单任务直接 ReAct

### ADR-022: Computer Use 优先使用 Accessibility API 而非纯视觉

**状态:** 已采纳

**背景:** Computer Use 可通过截屏+视觉识别或 Accessibility API 实现。

**决策:** 三层策略：API/Script → Accessibility → Vision，优先使用高效精确的方法。

**理由:**
- 纯视觉方案 Token 消耗巨大 (每次截屏约 1-2K tokens)，且易误操作
- Accessibility API 是 macOS 原生能力，精确且低开销
- Vision 作为通用兜底，处理无法通过 API 操作的第三方应用

### ADR-023: 信任等级按 Agent 独立计算

**状态:** 已采纳

**背景:** 信任等级可以是全局的（所有 Agent 共享）或 Per-Agent 的。

**决策:** Per-Agent 独立信任等级。

**理由:**
- 不同 Agent 能力范围不同，同一信任等级含义不同
- 用户可能信任"写作助手"的自主操作但不信任"开发者"的 Shell 执行
- Per-Agent 更符合"专业化分工"的产品理念

### ADR-024: 预测性主动基于记忆系统而非独立行为分析库

**状态:** 已采纳

**背景:** 行为模式可以存储在独立的分析数据库或复用现有记忆系统。

**决策:** 行为模式作为特殊类型的 Agent Memory 存储在现有记忆系统中。

**理由:**
- 复用三层记忆架构的衰减、召回、审计能力
- 避免引入新的存储引擎
- 行为模式本质上就是 Agent 对用户的"学习记忆"
