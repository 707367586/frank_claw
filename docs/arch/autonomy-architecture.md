## 1. AI 自主性发展现状分析

### 1.1 自主性演进阶梯

AI Agent 的自主性可以划分为 5 个层级，每个层级代表不同的能力边界：

```
Level 5: 完全自主 (Full Autonomy)
         自主设定目标、自主学习新能力、跨域迁移
         ◈ 当前技术尚未达到

Level 4: 协作自主 (Collaborative Autonomy)  ← ClawX 目标层级
         多 Agent 协作、层级规划与分解、自我反思与纠错
         自主发现并学习新工具、理解物理环境
         ◈ 2025-2026 前沿: Claude Computer Use, OpenAI Operator,
           Devin, Multi-Agent Systems (AutoGen/CrewAI)

Level 3: 工具自主 (Tool Autonomy)
         ReAct 推理循环、多步工具调用、上下文感知决策
         ◈ 2024-2025 主流: Claude Tool Use, GPT Actions, LangChain Agents

Level 2: 记忆自主 (Memory Autonomy)
         持久化记忆、个性化适配、跨会话连续性
         ◈ ClawX 已有完整设计 (三层记忆架构)

Level 1: 响应式 (Reactive)
         单轮问答、无状态、无工具调用
         ◈ 传统 Chatbot

当前 ClawX 架构位于 Level 2-3 之间，本文档将架构提升至 Level 4
```

## 3. ReAct 推理循环 (Reasoning + Acting)

### 3.1 设计原理

传统的 ClawX 对话流是线性的：`用户输入 → Prompt 组装 → LLM 调用 → 返回`。这限制了 Agent 处理需要多步骤推理的复杂任务。

ReAct 循环让 Agent 具备**迭代推理**能力：

```
┌──────────────────────────────────────────────────────────────────────┐
│                      ReAct Loop Architecture                          │
│                                                                       │
│  用户输入: "帮我调研竞品 X 的定价策略，并生成对比分析报告"              │
│      │                                                                │
│      ▼                                                                │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Iteration 1                                                     │ │
│  │  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌────────────────┐  │ │
│  │  │ Thought │──▶│ Action  │──▶│Observ.  │──▶│ Evaluation     │  │ │
│  │  │ 我需要先│   │ 使用    │   │ 搜索到  │   │ 信息不够完整   │  │ │
│  │  │ 搜索竞品│   │ web_    │   │ 3 篇文章│   │ 需要更多细节   │  │ │
│  │  │ 定价信息│   │ search  │   │ 关于X   │   │ → 继续迭代     │  │ │
│  │  └─────────┘   └─────────┘   └─────────┘   └────────────────┘  │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│      │                                                                │
│      ▼ (继续)                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Iteration 2                                                     │ │
│  │  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌────────────────┐  │ │
│  │  │ Thought │──▶│ Action  │──▶│Observ.  │──▶│ Evaluation     │  │ │
│  │  │ 需要读取│   │ read_   │   │ 获取到  │   │ 定价数据已充分 │  │ │
│  │  │ 文章详情│   │ document│   │ 详细定价│   │ 可以开始分析   │  │ │
│  │  │ 获取定价│   │ (×3)   │   │ 方案数据│   │ → 继续迭代     │  │ │
│  │  └─────────┘   └─────────┘   └─────────┘   └────────────────┘  │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│      │                                                                │
│      ▼ (继续)                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Iteration 3                                                     │ │
│  │  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌────────────────┐  │ │
│  │  │ Thought │──▶│ Action  │──▶│Observ.  │──▶│ Evaluation     │  │ │
│  │  │ 数据充分│   │ write_  │   │ 报告已  │   │ ✓ 任务完成     │  │ │
│  │  │ 生成对比│   │ document│   │ 生成完毕│   │ 返回最终结果   │  │ │

│  │  └─────────┘   └─────────┘   └─────────┘   └────────────────┘  │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│      │                                                                │
│      ▼                                                                │
│  最终输出: 竞品定价对比分析报告 (PDF) + 关键发现摘要                    │
└──────────────────────────────────────────────────────────────────────┘

### 3.2 ReAct 状态机

**核心状态流转：** Thinking → Acting → Observing → Evaluating → Completed | AwaitingConfirmation

| 状态 | 说明 |
|------|------|
| Thinking | LLM 分析当前状态，决定下一步 |
| Acting | 执行 LLM 决定的工具调用 |
| Observing | 收集工具执行结果 |
| Evaluating | 判断是否需要继续迭代 (Continue/Complete/NeedHumanInput/Error) |
| Completed | 返回最终结果和完整推理链 |
| AwaitingConfirmation | 高风险操作等待用户确认 |

每个状态携带迭代计数器。完成时输出推理链，每步记录思考过程、执行操作、观察结果、评估结论、耗时和 Token 消耗。

### 3.3 安全围栏 (Safety Guardrails)

```
ReAct 循环安全约束:

┌──────────────────────────────────────────────────────────┐
│                    Safety Guardrails                       │
│                                                           │
│  1. 最大迭代次数                                          │
│     ├── 默认: max_iterations = 10                        │
│     ├── 用户可配置: 1-25                                  │
│     └── 每次迭代消耗 Token 追踪，超过预算自动终止          │
│                                                           │
│  2. 操作风险分级                                          │
│     ├── Low: 纯读取操作 (search, read) → 自动执行         │
│     ├── Medium: 受限写入 (write_to_workspace) → 自动执行  │
│     ├── High: 外部交互 (send_message, api_call) → 需确认  │
│     └── Critical: 系统操作 (exec_shell, delete) → 必须确认│
│                                                           │
│  3. 循环检测                                              │
│     ├── 连续 3 次执行相同 Action → 告警并暂停              │
│     ├── 检测到无进展 (observation 重复) → 请求用户指导     │
│     └── Token 总消耗 > budget * 0.8 → 警告并建议总结      │
│                                                           │
│  4. 实时透明                                              │
│     ├── 每步 Thought 实时流式展示给用户                    │
│     ├── 用户可随时按 Esc 中断循环                          │
│     └── 中断后保留已完成步骤的结果                         │
└──────────────────────────────────────────────────────────┘
```

## 4. 自我反思与纠错 (Self-Reflection)

### 4.1 设计原理

自我反思是最能带来 "Aha Moment" 的能力——用户看到 Agent 不仅给出答案，还能自己检查、发现问题、并主动修正。这种类人行为大幅提升用户信任感。

### 4.2 反思架构

```
┌──────────────────────────────────────────────────────────────────────┐
│                     Self-Reflection Architecture                       │
│                                                                       │
│  ┌─────────────┐                                                      │
│  │ Agent 产出  │                                                      │
│  │ 初始响应    │                                                      │
│  └──────┬──────┘                                                      │
│         │                                                              │
│         ▼                                                              │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Reflection Engine (反思引擎)                                     │ │
│  │                                                                   │ │
│  │  ┌──────────────────┐  ┌──────────────────┐                      │ │
│  │  │ 质量检查器       │  │ 一致性检查器      │                      │ │
│  │  │ (Quality Check)  │  │ (Consistency)     │                      │ │
│  │  │                  │  │                   │                      │ │
│  │  │ • 完整性: 是否   │  │ • 与记忆一致?     │                      │ │
│  │  │   回答了所有问题 │  │ • 与知识库一致?   │                      │ │
│  │  │ • 准确性: 数据   │  │ • 与之前回答一致? │                      │ │
│  │  │   是否可验证     │  │ • 逻辑自洽?       │                      │ │
│  │  │ • 格式: 符合     │  │                   │                      │ │
│  │  │   用户偏好?      │  │                   │                      │ │
│  │  └────────┬─────────┘  └────────┬──────────┘                      │ │
│  │           │                      │                                 │ │
│  │           └──────────┬───────────┘                                 │ │
│  │                      ▼                                             │ │
│  │           ┌──────────────────────┐                                 │ │
│  │           │  反思决策            │                                 │ │
│  │           │                     │                                 │ │
│  │           │  confidence >= 0.85 │─── ✅ 直接返回                   │ │
│  │           │  confidence 0.5~0.85│─── 🔄 自动修正后返回            │ │
│  │           │  confidence < 0.5   │─── ⚠️ 告知用户不确定性          │ │
│  │           └──────────────────────┘                                 │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│         │                                                              │
│         ▼                                                              │
│  ┌─────────────┐                                                      │
│  │ 最终响应    │  可能包含:                                           │
│  │ (改进版)    │  "我重新检查了一下，发现之前的数据有误，已修正..."    │
│  └─────────────┘                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 4.3 反思类型

**检查维度：**

| 维度 | 检查内容 |
|------|---------|
| 质量检查 | 完整性、准确性 (可通过 KB 验证)、格式匹配度 |
| 一致性检查 | 与记忆一致、与知识库一致、前后逻辑自洽、矛盾点检测 |
| 安全检查 | 敏感数据、有害内容、准则遵循 |
| 任务完成度 | 原始意图匹配、已覆盖方面、遗漏方面、完成比例 |

**修正策略：** 补充遗漏 │ 修正错误 │ 重新组织格式 │ 完全重做 │ 请求用户澄清

反思引擎输出综合置信度评分，并根据评分决定是否触发修正。

### 4.4 反思触发策略

并非每次都触发反思（LLM 开销过大），采用智能触发：

| 触发条件 | 反思深度 | 说明 |
|---------|---------|------|
| 用户明确要求高质量 ("仔细"、"确保准确") | Full | 全量检查 |
| Agent 输出包含数据/数字/事实声明 | Accuracy | 数据验证 |
| 输出长度 > 500 字 | Completeness | 完整性检查 |
| 涉及用户偏好的任务 (写作风格等) | Consistency | 记忆一致性 |
| ReAct 循环中每次关键决策点 | Quick | 快速自检 |
| 对话轮次 > 5 且任务复杂 | TaskCompletion | 任务完成度 |
| 默认 (简单问答) | None | 不触发反思 |

### 4.5 Aha Moment 示例

```
用户: 帮我用 Python 写一个快速排序

Agent (初始): [生成代码]

Agent (反思后):
"我检查了一下刚才写的代码，发现两个问题：
 1. 没有处理空列表的边界情况
 2. 根据你之前的偏好（记忆召回），你喜欢 type hints 风格

 已自动修正，这是改进后的版本：

 ```python
 def quicksort(arr: list[int]) -> list[int]:
     if len(arr) <= 1:  # 修正: 处理边界
         return arr
     ...
 ```"
```

## 5. 层级任务规划 (Hierarchical Planning)

### 5.1 设计原理

对于复杂任务，Agent 需要将目标分解为可执行的子任务树，并在执行过程中根据结果动态调整计划。

### 5.2 规划架构

```
┌──────────────────────────────────────────────────────────────────────┐
│                   Hierarchical Planner Architecture                    │
│                                                                       │
│  用户: "帮我准备下周一的技术分享演讲"                                   │
│      │                                                                │
│      ▼                                                                │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Phase 1: 意图理解与目标分解                                      │ │

### 5.3 核心概念

**任务计划 (TaskPlan)：** 包含顶层目标和根任务节点树，计划状态可为 Draft → Approved → InProgress → Paused → Completed/Failed。

**任务节点 (TaskNode)：** 树形结构，每个节点包含：
- 标题、描述、前置依赖列表、子任务列表
- 所需工具、预估 Token 消耗、是否可与兄弟节点并行执行
- 状态：Pending / Running / WaitingForUser / Completed / Failed / Skipped / Replanned

## 6. 多 Agent 协作编排 (Multi-Agent Orchestration)

### 6.1 设计原理

ClawX 已支持多 Agent 配置，但当前架构中 Agent 之间是完全隔离的（循环守卫 L9 防止 Agent 间调用）。本设计引入**受控的结构化协作**，允许 Agent 在明确的协作协议下互相委托任务。

> **关键区别**：不是无限制的 Agent A ↔ B 互调（仍然危险），而是通过 **Orchestrator 模式** 实现单向、可控的委托。

### 6.2 协作模式

```
┌──────────────────────────────────────────────────────────────────────┐
│            Multi-Agent Orchestration Patterns                         │
│                                                                       │
│  Pattern 1: 委托模式 (Delegation)                                     │
│  ─────────────────────────────────                                    │
│  适用: 一个 Agent 需要另一个 Agent 的专业能力                          │
│                                                                       │
│  ┌──────────┐   委托任务    ┌──────────┐                             │
│  │ PM Agent │──────────────▶│ Dev Agent│                             │
│  │ (发起方) │   "写个脚本   │ (执行方) │                             │
│  │          │    处理数据"  │          │                             │
│  │          │◀──────────────│          │                             │
│  └──────────┘   返回结果    └──────────┘                             │
│                                                                       │
│  约束: 委托深度 <= 2 (A→B→C 可以, A→B→C→D 禁止)                     │
│  约束: 发起方必须有 delegate 权限                                     │
│  约束: 被委托方只能访问委托消息上下文，不能访问发起方的完整记忆        │
│                                                                       │
│                                                                       │
│  Pattern 2: 协商模式 (Negotiation)                                    │
│  ─────────────────────────────────                                    │
│  适用: 需要多个 Agent 对同一问题提供不同视角                           │
│                                                                       │
│        ┌──────────┐                                                   │
│        │   User   │ "这个架构方案怎么样？"                             │
│        └────┬─────┘                                                   │
│             │ 广播                                                     │
│      ┌──────┼──────┐                                                  │
│      ▼      ▼      ▼                                                  │
│  ┌──────┐┌──────┐┌──────┐                                            │
│  │ Dev  ││ Sec  ││ PM   │                                            │
│  │Agent ││Agent ││Agent │                                            │
│  │      ││      ││      │                                            │
│  │"性能 ││"安全 ││"工期 │                                            │
│  │ 不错"││ 有风 ││ 偏长"│                                            │
│  │      ││ 险点"││      │                                            │
│  └──┬───┘└──┬───┘└──┬───┘                                            │
│     └───────┼───────┘                                                 │
│             ▼                                                         │
│       ┌───────────┐                                                   │
│       │ 综合意见  │  → 返回用户                                       │
│       └───────────┘                                                   │
│                                                                       │
│                                                                       │
│  Pattern 3: 流水线模式 (Pipeline)                                     │
│  ──────────────────────────────                                       │
│  适用: 任务可分解为顺序执行的专业步骤                                  │
│                                                                       │
│  ┌──────────┐  数据  ┌──────────┐  分析  ┌──────────┐  报告          │
│  │ Research │──────▶│ Analyst  │──────▶│ Writer   │──────▶ 最终产物  │
│  │ Agent    │ 原始  │ Agent    │ 结论  │ Agent    │                   │
│  │ 收集数据 │ 数据  │ 分析数据 │       │ 生成报告 │                   │
│  └──────────┘       └──────────┘       └──────────┘                   │
│                                                                       │
│  约束: 单向流动，无回路                                               │
│  约束: 每步输出即下步输入，格式协议明确                               │
└──────────────────────────────────────────────────────────────────────┘
```

### 6.3 协作协议

**协作请求核心要素：**

| 要素 | 说明 |
|------|------|
| 发起方/被委托方 | Agent ID 对 |
| 协作模式 | Delegation (1:1) / Negotiation (多方) / Pipeline (顺序) |
| 委托上下文 | 最小化上下文——仅包含任务描述、相关数据片段、期望输出格式、Token 预算；不暴露发起方完整记忆 |
| 委托约束 | 最大委托深度 (默认 2)、最大迭代次数、超时时间、可用工具白名单、是否需用户批准 |

### 6.4 与现有安全架构的集成

```
协作安全层 (扩展 L9 循环守卫):

现有 L9: 检测 Agent A ↔ B 乒乓模式，直接拒绝
升级 L9':
  ├── 委托链追踪: 每个 DelegationRequest 携带 chain_hash
  ├── 深度限制: depth > max_depth → 拒绝
  ├── 环路检测: 调用链中出现同一 Agent → 拒绝
  ├── 频率限制: 单 Agent 每小时委托次数上限
  ├── 上下文隔离: 被委托方只看到 DelegationContext，不看到完整记忆
  └── 审计日志: 所有协作行为记入 audit_log，可追溯
```

│                                                           │
│  优势: 快速、精确、不受 UI 变化影响                        │

│                                                           │

### 8.4 Computer Use 安全模型

```
Computer Use 是最高风险的自主能力，必须严格约束:

┌──────────────────────────────────────────────────────────┐
│                Computer Use Safety                        │
│                                                           │
│  1. 权限要求                                              │
│     ├── macOS Accessibility 权限 (系统级)                  │
│     ├── Screen Recording 权限 (系统级)                    │
│     ├── ClawX Computer Use 能力开关 (用户显式启用)        │
│     └── Per-Agent Computer Use 权限 (每个 Agent 独立)     │
│                                                           │
│  2. 操作约束                                              │
│     ├── 白名单应用: 只能操作用户授权的应用                 │
│     ├── 禁止区域: Terminal、系统设置、密钥链 等默认禁止    │
│     ├── 操作限速: 最快 500ms/操作，防止快速误操作          │
│     ├── 关键操作确认: 文件删除、发送消息、支付 等必须确认  │
│     └── 撤销能力: 每步操作前截屏保存，支持 Cmd+Z 撤销    │
│                                                           │
│  3. 透明性                                                │
│     ├── 操作区域高亮: Agent 操作时，屏幕上高亮操作区域    │
│     ├── 操作日志: 每次点击/输入记录坐标、目标、截图       │
│     ├── 实时观察: GUI 面板实时展示 Agent 正在做什么        │
│     └── 紧急停止: 全局热键 (如 Cmd+Shift+Esc) 立即停止   │
│                                                           │
│  4. 信任等级关联                                          │
│     ├── Level 0: 禁用 Computer Use                        │
│     ├── Level 1: 只读 (截屏分析，不操作)                  │
│     ├── Level 2: 受限操作 (白名单应用，每步确认)          │
│     ├── Level 3: 自主操作 (白名单应用，高风险确认)        │
│     └── Level 4: 完全自主 (仅禁止区域限制)                │
└──────────────────────────────────────────────────────────┘
```

### 8.5 Aha Moment 示例

```

  7. 完成后截屏 → 展示对比

→ Aha Moment: "它居然能直接操作我的 Figma！"

## 9. 信任渐进系统 (Progressive Trust)

### 9.1 设计原理

自主性越高，风险越大。信任渐进系统让 Agent 通过成功执行逐步积累用户信任，用信任等级动态调整 Agent 的自主权限边界。

### 9.2 信任等级模型

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Progressive Trust Model                             │
│                                                                       │
│  Trust Level 0: 新手 (Novice)                                        │
│  ─────────────────────────────                                        │
│  • 所有工具调用需用户确认                                              │
│  • ReAct 循环限制 3 次迭代                                            │
│  • 无 Computer Use                                                    │
│  • 无 Agent 间委托                                                    │
│  • 触发条件: 新创建的 Agent                                           │
│                                                                       │
│  Trust Level 1: 学徒 (Apprentice)                                     │
│  ─────────────────────────────────                                    │
│  • 低风险操作自动执行 (读取、搜索)                                    │
│  • ReAct 循环限制 5 次迭代                                            │
│  • Computer Use: 只读模式                                             │
│  • 无 Agent 间委托                                                    │
│  • 升级条件: 成功执行 10 个任务，0 次严重错误                          │
│                                                                       │
│  Trust Level 2: 助手 (Assistant)                                      │
│  ────────────────────────────────                                     │
│  • 中等风险操作自动执行 (工作区写入)                                   │
│  • ReAct 循环限制 10 次迭代                                           │
│  • Computer Use: 白名单应用受限操作                                   │
│  • 可发起 Agent 间委托 (需确认)                                       │
│  • 升级条件: 成功执行 50 个任务，< 2 次用户回滚                        │
│                                                                       │
│  Trust Level 3: 伙伴 (Partner)                                        │
│  ─────────────────────────────                                        │
│  • 仅高风险操作需确认 (外部通信、系统操作)                             │
│  • ReAct 循环限制 15 次迭代                                           │
│  • Computer Use: 白名单应用自主操作                                   │
│  • 可发起 Agent 间委托 (自动)                                         │
│  • 可主动发起建议性任务                                                │
│  • 升级条件: 成功执行 200 个任务，用户满意度 > 90%                     │
│                                                                       │
│  Trust Level 4: 专家 (Expert)                                         │
│  ──────────────────────────────                                       │
│  • 仅关键操作需确认 (删除、支付、外部消息)                             │
│  • ReAct 循环限制 25 次迭代                                           │
│  • Computer Use: 完全自主 (禁止区域除外)                               │
│  • 完全自主的 Agent 间协作                                            │
│  • 可主动执行预授权任务                                                │
│  • 升级条件: 长期良好表现，用户手动提升                                │
└──────────────────────────────────────────────────────────────────────┘
```

### 9.3 信任评分算法

**评分公式：**

```
score = success_rate × 60 + feedback_bonus (max 20) - rollback_penalty (×5) - correction_penalty (×2)
score *= recency_factor (严重事件 30 天后影响消退)
最终 score 取值 [0, 100]
```

**输入维度：** 成功/总任务数、用户回滚次数、用户纠正次数、正面/负面反馈数、最近严重事件时间。

### 9.4 信任降级 (Demotion)

信任可以降级，严重错误导致即时降级：

| 事件 | 影响 |
|------|------|
| 用户回滚 Agent 操作 | score -= 5 |
| 用户纠正 Agent 输出 | score -= 2 |
| Agent 执行出错 (可恢复) | score -= 3 |
| Agent 执行出错 (不可恢复) | 立即降级一层 |
| 安全事件 (DLP 触发等) | 立即降级到 Level 0 |
| 用户明确表示不满 | score -= 10 |

## 10. 预测性主动 (Predictive Proactivity)

### 10.1 设计原理

超越 PRD 中简单的 Cron 定时触发，实现**基于用户行为模式的预测性主动**——Agent 学习用户的工作节奏，在用户需要之前就准备好。

### 10.2 预测引擎

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Predictive Proactivity Engine                       │
│                                                                       │
│  数据源:                                                              │
│  ├── 对话历史: 用户通常在什么时间问什么类型的问题                      │
│  ├── 任务记录: 用户的周期性任务模式 (每周一写周报, 每天看新闻)        │
│  ├── 日历事件: 通过 Calendar API 获取即将到来的事件                   │
│  ├── 文件活动: FSEvents 监控文件操作模式                              │
│  └── 系统状态: 磁盘空间、网络状态等                                   │
│                                                                       │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Pattern Recognition (模式识别)                                   │ │
│  │                                                                   │ │
│  │  1. 时间模式: "用户每周五下午 4 点请求写周报"                     │ │
│  │     → 周五 3:30 开始收集本周任务数据                              │ │
│  │                                                                   │ │
│  │  2. 序列模式: "用户总是先搜索论文，再请求总结"                    │ │
│  │     → 检测到搜索行为后，预加载总结模板                            │ │
│  │                                                                   │ │
│  │  3. 上下文模式: "用户每次会议前 30 分钟请求准备材料"              │ │
│  │     → 检测到日历事件后，主动准备相关材料                          │ │
│  │                                                                   │ │
│  │  4. 异常模式: "磁盘空间低于 10%"                                 │ │
│  │     → 主动分析可清理内容并建议                                    │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                                                                       │
│  ┌──────────────────────────────────────────────────────────────────┐ │
│  │  Action Selection (行动选择)                                      │ │
│  │                                                                   │ │
│  │  预测置信度 >= 0.8 且 行动风险 = Low:                             │ │
│  │    → 自动执行 (如预缓存数据、生成草稿)                            │ │
│  │                                                                   │ │
│  │  预测置信度 >= 0.6 且 行动风险 = Medium:                          │ │
│  │    → 建议性通知 ("需要我帮你准备周报吗？")                        │ │
│  │                                                                   │ │
│  │  预测置信度 < 0.6:                                                │ │
│  │    → 不行动，继续观察积累                                         │ │
│  └──────────────────────────────────────────────────────────────────┘ │
│                                                                       │
│  反馈闭环:                                                            │
│  ├── 用户接受建议 → 强化该模式权重                                    │
│  ├── 用户忽略建议 → 降低该模式置信度                                  │
│  ├── 用户明确拒绝 → 记录为负反馈，衰减模式                           │
│  └── 用户说"不再提醒" → 永久抑制该模式                               │
└──────────────────────────────────────────────────────────────────────┘
```

### 10.3 行为模式数据模型

**行为模式核心属性：** 模式类型、自然语言描述、置信度 (0-1)、触发条件、建议行动、观察次数、接受/拒绝次数、是否被永久抑制。

**模式类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| 时间模式 | 学习到的 Cron 表达式 + 时间容差 | "用户每周五下午 4 点写周报" |
| 序列模式 | 前置行为序列 → 后续行为 | "搜索论文后总是请求总结" |
| 上下文模式 | 上下文信号触发 | "日历事件前 30 分钟准备材料" |
| 异常模式 | 指标超阈值 | "磁盘空间低于 10%" |

**主动行动类型：** 静默预处理 (用户不感知) │ 建议性通知 (温和提醒) │ 直接执行 (高置信度+低风险)

## 12. 自主性 Aha Moment 场景设计

### 12.1 高价值场景一览

| # | 场景 | 触发 | Agent 自主行为 | Aha Moment |
|---|------|------|--------------|-----------|
| 1 | **自主纠错** | 用户要求写代码 | Agent 写完代码后自检发现 bug，主动修复并说明 | "它居然能自己发现自己的错误！" |
| 2 | **主动准备** | 日历显示 30 分钟后有会议 | Agent 自动整理相关资料并推送 | "它知道我马上要开会！" |
| 3 | **跨 Agent 协作** | 用户要求 PM Agent 完成复杂项目 | PM 自动委托任务给 Dev 和 Writer Agent | "它们居然自己分工协作！" |
| 4 | **操作电脑** | "帮我把这些照片导入 Lightroom" | Agent 打开 Lightroom，操作导入界面 | "它居然在帮我操作软件！" |
| 5 | **学习偏好** | 用户多次修改 Agent 的输出格式 | Agent 主动说"我注意到你偏好 X 格式，已记住" | "它学会了我的习惯！" |
| 6 | **深度研究** | "帮我深入调研 XX 技术" | Agent 自主执行 8 步 ReAct：搜索→阅读→分析→补充→对比→总结→生成报告 | "它像真的研究员一样工作！" |
| 7 | **预判需求** | 用户打开了项目代码文件 | Agent 检测到文件变更，主动提供相关文档和最近的修改记录 | "它知道我接下来要做什么！" |
| 8 | **异常告警** | 磁盘空间突然降到 5% | Agent 分析原因，发现是 Docker 镜像占用，建议清理 | "它比我更早发现问题！" |

### 12.2 Aha Moment 度量

| 指标 | 目标 | 说明 |
|------|------|------|
| 首次 Aha Moment 时间 | < 10 分钟使用 | 新用户应在 10 分钟内体验到至少 1 个 Aha Moment |
| 日均 Aha Moment | >= 2 次 | 活跃用户每天至少感受 2 次惊喜 |
| 自主纠错触发率 | 输出的 15-25% | 太低没感知，太高说明初始质量差 |
| 预测性建议采纳率 | >= 60% | 低于 60% 说明预测不准，需调优 |
| 信任等级提升速率 | 1 周内达到 Level 2 | 正常使用 1 周后应有足够自主权限 |
