# ClawX 自主性架构设计 (Autonomy Architecture)

**版本:** 3.0
**日期:** 2026年3月18日
**对应 PRD:** v2.0 §2.8 主动式 Agent
**对应架构:** v4.1 clawx-runtime
**关联 ADR:** ADR-015, ADR-020, ADR-027, ADR-028, ADR-030

---

## 1. 设计目标与边界

### 1.1 问题定义

当前 ClawX 的 Agent 是被动的：用户发一条消息，Agent 回一条消息。这限制了两类核心场景：

1. **多步执行**：用户说"帮我查最近的 AI Safety 论文，整理摘要，生成报告"——Agent 需要自主完成搜索、筛选、摘要、排版等多个步骤，而不是每步都等用户确认。
2. **主动触达**：用户说"每天早上 8:30 帮我检查 arXiv 新论文"——Agent 需要在用户不在场时自主执行任务，并将结果推送给用户。

自主性架构的目标：**让 Agent 在安全围栏内完成多步任务，并能被定时、事件或上下文触发主动执行。**

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| 实用优先 | 先解决用户真实场景，不追求学术完备性 |
| 受控执行 | 所有自主行为在安全围栏内，用户可随时中断、暂停、回滚 |
| 可解释 | 每个执行步骤对用户透明，推理链可追溯 |
| 渐进开放 | 权限随使用积累逐步放宽，而不是一刀切 |
| 安全上限 | 自主性不能越过 `clawx-security` 的安全边界 (ADR-015) |
| 最小增量 | 集成在 `clawx-runtime` 子模块 (ADR-027)，不新增 Crate |

### 1.3 非目标

- 不追求完全自主——Agent 不自主设定高层目标，始终服务于用户意图。

### 1.4 能力与阶段对齐

> **阶段说明 (ADR-020)：** v0.1 只有基础护栏；v0.2 引入完整自主执行能力。

| 能力 | 阶段 | 说明 |
|------|------|------|
| 基础护栏 | **v0.1** | 最大迭代次数、Token 预算限制、循环检测 |
| 多步工具调用 | **v0.2** | 多步工具调用 + 中间推理 + 安全围栏 |
| 主动任务系统 | **v0.2** | 定时/事件触发 → 自主执行 → 结果推送 |
| 权限渐进 | **v0.2** | Per-Agent 权限等级，动态调整确认范围 |
| 任务管理 | **v0.2** | 创建/暂停/恢复/取消/查看执行历史 |

## 2. 架构总览

### 2.1 核心概念

自主性架构引入两个核心概念：

- **Execution（执行）**：Agent 接收一个目标，通过多步工具调用自主完成，对应交互式场景。
- **Task（主动任务）**：可被定时或事件触发的执行单元，在后台运行，结果推送给用户。

两者的关系：Task 是 Execution 的调度包装——Task 定义"何时触发"和"如何通知"，Execution 负责"如何完成"。

### 2.2 在 Runtime 中的位置

```
┌────────────────────────────────────────────────────────────────┐
│                        clawx-runtime                           │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │              Autonomy Module (v0.2)                       │  │
│  │                                                           │  │
│  │  ┌─────────────────┐    ┌──────────────────────────────┐  │  │
│  │  │  Executor        │    │  Task Manager                │  │  │
│  │  │  多步工具调用循环 │    │  主动任务生命周期管理        │  │  │
│  │  └────────┬─────────┘    └──────────┬───────────────────┘  │  │
│  │           │                         │                      │  │
│  │  ┌────────┴─────────────────────────┴───────────────────┐  │  │
│  │  │              Permission Gate                          │  │
│  │  │   权限等级 │ 操作风险评估 │ 确认/自动执行决策          │  │  │
│  │  └──────────────────────────────────────────────────────┘  │  │
│  └──────────────────────────────────────────────────────────┘  │

         │                    │                    │
         ▼                    ▼                    ▼
   clawx-scheduler     clawx-channel        clawx-security
   (定时/事件触发)     (结果推送)           (权限校验/DLP)
```

### 2.3 两种执行路径

**路径 A：交互式执行（用户在场）**

```
User Input → clawx-runtime
  → 安全检查 + 记忆/知识召回
  → 意图评估
     ├── 简单问答 → 传统线性流程 (Prompt → LLM → Response)
     └── 多步任务 → Executor 多步工具调用循环
  → DLP 扫描
  → Response → User (流式展示每步进展)
```

**路径 B：主动任务执行（用户不在场）**

```
clawx-scheduler (Cron / Event)
  → Task Manager: 加载任务定义
  → Executor: 执行多步工具调用
  → 持久化执行结果
  → 推送通知 → clawx-channel (IM) / macOS Notification / 文件写入
```

## 3. Executor：多步工具调用循环

### 3.1 核心思路

Executor 将 Agent 的执行从"单次 LLM → 单次工具调用"扩展为 **多轮工具调用循环**：LLM 分析当前状态 → 决定下一步行动 → 执行工具 → 观察结果 → 决定是否继续。

```
              ┌──────────────────┐
              │    LLM 推理       │◄──────────────────┐
              │  分析状态         │                    │
              │  决定下一步       │                    │
              └────────┬─────────┘                    │
                       │                              │
            ┌──────────┴──────────┐                   │
            │                     │                   │
       需要工具调用          直接输出答案              │
            │                     │                   │
            ▼                     ▼                   │
   ┌─────────────────┐    ┌──────────┐               │
   │ Permission Gate  │    │  Done    │               │
   │ 风险评估+权限检查│    └──────────┘               │
   └───┬────┬────────┘                                │
  自动放行  需确认                                     │
       │    ▼                                         │
       │  等待用户确认 → 批准/拒绝                     │
       ▼                                              │
   ┌──────────────────┐                               │
   │  执行工具调用     │                               │
   └────────┬─────────┘                               │
            ▼                                         │
   ┌──────────────────┐    继续                       │
   │  收集结果         │──────────────────────────────┘
   │  评估进展         │
   └───┬──────────────┘
       │ 异常/超限
       ▼
   ┌──────────────┐
   │  终止并汇报   │
   └──────────────┘
```

### 3.2 安全围栏

| 约束 | 规则 | 触发行为 |
|------|------|---------|
| 步数上限 | 默认 10 步，用户可配 1-25 | 超出后终止，返回已有结果 |
| Token 预算 | 持续追踪，80% 时警告，100% 时终止 | 终止并汇报已消耗 Token |
| 循环检测 | 连续 3 次相同工具调用或相似结果 | 暂停，请求用户指导 |
| 超时 | 默认 5 分钟（主动任务可配 30 分钟） | 终止并保存中间结果 |
| 实时透明 | 每步 thought 流式展示（交互式场景） | 用户可随时按 Esc 中断 |
| 权限检查 | 每次工具调用经过 Permission Gate | 见 §5 权限渐进 |

### 3.3 意图评估

进入 Executor 前，Runtime 先评估是否需要多步执行。v0.2 通过 LLM 单次调用判断——如果用户意图需要 2 个以上步骤或涉及多个工具，走 Executor；否则走传统线性流程。判断逻辑内嵌在 System Prompt 中，不需要独立模块。

## 4. Task Manager：主动任务系统

### 4.1 核心思路

Task Manager 管理"主动任务"的完整生命周期——从创建、触发、执行到通知。它是 PRD §2.8 主动式 Agent 的架构落地。

Task Manager 不自己做调度——定时触发和事件监听由 `clawx-scheduler` 负责。Task Manager 的职责是：

1. 存储任务定义（目标、配置、通知方式）
2. 被 Scheduler 触发时，调用 Executor 执行
3. 持久化执行结果
4. 推送通知
5. 收集用户反馈（采纳/忽略/拒绝）

### 4.2 任务定义

每个主动任务包含以下要素：

| 字段 | 说明 |
|------|------|
| 任务名称 | 用户可见的名称 |
| 关联 Agent | 由哪个 Agent 执行 |
| 执行目标 | 传给 Executor 的 goal 文本 |
| 触发规则 | Cron 表达式 / 事件类型 / 一次性定时 |
| 通知方式 | 桌面通知 / IM 渠道 / 文件写入（可多选） |
| 执行配置 | 可覆盖 Agent 默认的步数上限、超时等 |
| 创建方式 | 对话创建 / 面板手动创建 / Agent 建议 |
| 状态 | 活跃 / 已暂停 / 执行中 / 错误 / 已归档 |

**触发类型：**

| 触发类型 | 描述 | 示例 |
|---------|------|------|
| 定时任务 | Cron 表达式或自然语言 | "每天早上 8:30"、"每周一下午 3 点" |
| 事件驱动 | 系统事件触发 | 文件变更、磁盘空间 < 10% |
| 一次性 | 指定时间执行一次 | "明天下午 3 点提醒我开会" |

### 4.3 对话创建任务

用户在对话中用自然语言创建定时任务（PRD §2.8.3）：

```
用户: "帮我每天早上 8 点检查 arXiv 上 AI Safety 的新论文，整理摘要发飞书"

  → LLM 解析:
    触发: Cron "0 8 * * *" (每天早上 8 点)
    目标: "检查 arXiv 上 AI Safety 相关新论文，整理摘要"
    通知: 飞书渠道推送

  → Agent 回复确认:
    "已创建定时任务：每天早上 8:00 检查 arXiv AI Safety 新论文，结果通过飞书推送。"

  → Task 持久化 → clawx-scheduler 注册 Cron
```

### 4.4 执行历史与反馈

每次任务执行产生一条历史记录，包含：触发时间、执行状态（成功/失败/超时/取消）、执行结果摘要、步骤数和 Token 消耗。

用户可对每次执行结果给出反馈：

| 反馈类型 | 效果 |
|---------|------|
| 采纳 | 正常记录 |
| 忽略 | 连续 3 次忽略 → 自动暂停，通知用户确认是否继续 |
| 拒绝 | 记录原因，计入负反馈率 |
| 不再提醒 | 任务自动归档，不再触发 |
| 降低频率 | 触发间隔翻倍 |

**负反馈率 = (拒绝 + 不再提醒) / 总执行次数**，PRD 要求持续低于 15%。超过时 GUI 标记警告，建议用户调整或关闭。

### 4.5 与 clawx-scheduler 的职责划分

| 职责 | 归属 |
|------|------|
| Cron 解析与定时触发 | clawx-scheduler |
| 事件监听与条件匹配 | clawx-scheduler |
| 任务定义存储与管理 | Task Manager |
| 调用 Executor 执行 | Task Manager |
| 结果持久化与通知推送 | Task Manager → clawx-channel |
| 反馈收集与抑制策略 | Task Manager |

## 5. Permission Gate：权限渐进系统

### 5.1 核心思路

新建的 Agent 做什么都要用户确认很烦，但用了一段时间的 Agent 应该可以自动执行低风险操作。Permission Gate 实现 **Per-Agent 权限等级** (ADR-030)——等级越高，自动放行的操作范围越大。

### 5.2 权限等级

| 等级 | 名称 | 自动放行 | 需确认 | 达到条件 |
|------|------|---------|--------|---------|
| L0 | 受限 | 无 | 所有工具调用 | 新建 Agent 默认 |
| L1 | 基础 | 读取类（文件读、检索、搜索） | 写入、发送、执行 | 成功 10 个任务 |
| L2 | 标准 | + 工作区内文件写入 | 发送消息、API 调用、Shell | 成功 50 个任务，无严重错误 |
| L3 | 信任 | + 发送 IM 消息、外部 API | Shell、文件删除、系统操作 | 用户手动提升 |

### 5.3 操作风险分级

| 风险 | 典型操作 | L0 | L1 | L2 | L3 |
|------|---------|----|----|----|----|
| Read | 文件读取、知识库检索、网页搜索 | 确认 | 自动 | 自动 | 自动 |
| Write | 工作区内写入文件、创建目录 | 确认 | 确认 | 自动 | 自动 |
| Send | 发送 IM 消息、外部 API 请求 | 确认 | 确认 | 确认 | 自动 |
| Danger | Shell 命令、文件删除、系统配置 | 确认 | 确认 | 确认 | 确认 |

### 5.4 关键约束

- L3 必须用户手动提升，不自动升级。
- Danger 操作在任何等级下都需要确认。
- 安全事件（DLP 触发、权限越界）立即降级到 L0。
- L0 的 Agent 不能创建主动任务。

## 6. 与安全架构的集成

### 6.1 安全边界

自主执行受 `clawx-security` 纵深防御约束，不能越过安全上限 (ADR-015)：

| 安全层 | 与自主性的交互 |
|--------|-------------|
| L2 (分级执行) | Executor 中的工具调用仍走 T1/T2/T3 分级执行 |
| L4 (声明式权限) | Permission Gate 在 L4 边界内再叠加渐进式控制 |
| L5 (DLP) | Executor 每步工具返回结果都经 DLP 扫描 |
| L7 (路径隔离) | 文件操作受工作区边界约束 |
| L9 (循环守卫) | 扩展为支持 Executor 的步数上限和循环检测 |
| L12 (审计日志) | 执行步骤、任务触发、权限变更全部记入审计 |

### 6.2 L9 循环守卫升级

v0.2 将 L9 从"Agent 间乒乓检测"扩展为完整的执行守卫：

- 步数计数器：超过 max_steps → 强制终止
- 循环检测：连续相同工具调用 → 暂停请求用户指导
- Token 预算守卫：超出 budget → 警告/终止
- 超时守卫：超过 timeout → 终止并保存中间结果

### 6.3 审计日志扩展

| 事件类型 | 记录内容 |
|---------|---------|
| `execution_step` | 步骤编号、thought 摘要、工具调用、结果摘要 |
| `execution_complete` | 总步数、总 Token、最终结果摘要 |
| `task_triggered` | 任务 ID、触发类型、触发时间 |
| `task_completed` | 任务 ID、执行状态、结果摘要 |
| `confirmation_required` | 待确认操作、风险等级 |
| `confirmation_result` | 用户批准/拒绝 |
| `permission_change` | 旧/新等级、触发事件 |
| `task_feedback` | 任务 ID、反馈类型 |

## 7. 数据存储

自主性相关数据存储在 SQLite 主数据库，包含以下表：

| 表 | 用途 | 关键字段 |
|----|------|---------|
| `permission_profiles` | 权限档案 | agent_id, level, total_tasks, successful_tasks, serious_errors |
| `tasks` | 主动任务定义 | agent_id, name, goal, trigger_type, trigger_config(JSON), notification(JSON), status |
| `task_runs` | 任务执行历史 | task_id, triggered_at, run_status, result_summary, feedback, tokens_used, steps_count |
| `permission_events` | 权限变更审计 | agent_id, event_type, old_level, new_level, detail(JSON) |

## 8. 用户体验

### 8.1 交互式多步执行

多步执行时 GUI 流式展示每步进展：

```
🔄 Step 1/10: 搜索 arXiv AI Safety 最新论文...
   > 找到 12 篇相关论文

🔄 Step 2/10: 筛选最相关的 5 篇...
   > 已选出 Top 5 论文

🔄 Step 3/10: 生成摘要报告...
   > [流式输出摘要内容...]

✅ 完成 (3 步, 2847 tokens, 12.3s)
```

用户可随时按 Esc 中断，中断后保留已完成步骤的结果。

高风险操作弹出确认窗口，展示：工具名、具体参数、风险等级、Agent 的推理原因。

### 8.2 任务管理面板

展示所有主动任务列表（PRD §2.8.3 要求）：

- 每个任务显示：名称、关联 Agent、触发规则、上次/下次执行时间、状态
- 操作按钮：暂停/恢复、编辑、查看历史、删除
- 历史详情页：每次执行的步骤、结果、用户反馈、反馈统计（采纳率/负反馈率）

每一次主动触达都展示触发来源、触发规则、执行时间和执行结果（PRD 验收标准）。

### 8.3 权限等级展示

Agent 详情页展示当前权限等级、升级进度（如"35/50 任务 → L2 标准"）、当前自动放行和需确认的操作范围，以及手动调整入口。

|------|------|
| 多步执行 | 在 10 步以内完成"搜索+分析+生成报告"类任务 |
| 安全围栏 | 超出步数/Token/超时时正确终止；循环检测准确 |
| 主动任务 | Cron 任务按时触发，结果正确推送到指定渠道 |
| 对话创建 | 用户自然语言 → 正确解析为任务定义 |
| 反馈机制 | "不再提醒"→ 归档；"降低频率"→ 间隔翻倍 |
| 权限升级 | 正常使用 1 周内达到 L1 |
| 性能 | Executor 单步 overhead < 200ms（不含 LLM） |
| 审计 | 所有执行步骤、任务触发、权限变更可追溯 |
| 负反馈率 | 主动任务负反馈率持续低于 15% (PRD 要求) |
